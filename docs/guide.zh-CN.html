Whet 使用指南

把 Claude Code 磨砺（whet）成一支完整的产品团队。

版本 0.3.2 MIT 许可 Claude Code 插件

开始Whet 是什么

Whet 是一个 Claude Code 插件，为你的每个项目装配一支「虚拟产品团队」，并附带让这支团队在真实项目上高效运转的四套工作流：

🧑‍💻 11 个专业 Agent

产品经理、架构师、UI/UX 设计师、前端 / 后端 / 移动端工程师、QA、运维、代码评审员、任务审核器（对抗式完成度审计）、轻量执行员（低成本跑小任务）。

📋 长程任务调度

跨会话的大任务全部状态落盘到批次台账，按依赖与冲突域串并行派发，执行与审核分离，一次启动、全程零打断地跑完。

🎚️ 自动模型选择

平台无关的 T1–T4 能力档位抽象：计划里只写档位，运行时再映射到实际可用模型。从最低够用档起步，凭证据升档，绝不硬编码模型名。

💰 Token 消耗优化

会话内手法（输入最小化、缓存友好、输出纪律）+ 项目级四层诊断，并划定质量红线——损害判断质量的「节省」一律否决。

另外还有规格驱动开发（需求 → 设计 → 任务的三段门控流程）和三个安全 / 会话钩子。所有能力都以 Markdown + JSON 交付——没有构建步骤，没有运行时依赖。

名字的由来 whet /wɛt/ 意为「磨砺（刀刃）」。Claude Code 本身已是利器，Whet 负责把它磨得更锋利、更省、更有章法。

开始安装与更新

方式一：从插件市场安装（推荐）

# 在 Claude Code 会话中执行
/plugin marketplace add mufasa007/whet
/plugin install whet@whet

方式二：克隆后作为本地插件加载

git clone https://github.com/mufasa007/whet.git
claude --plugin-dir ./whet

更新

/plugin marketplace update whet   # 刷新市场清单
/plugin update whet@whet          # 更新已安装插件
更新须知 更新在新会话中生效。你项目里的 .whet/plan/** 台账与 spec/** 规格文档属于用户数据，保存在你自己的仓库中，插件更新永远不会改动它们。破坏性版本（MAJOR）必附迁移说明，见 CHANGELOG.md。

验证安装

  • 输入 /agents —— 应能看到 architect、backend-dev 等 11 个 whet agent；
  • 输入 /help —— 应能看到 /whet:spec、/whet:plan 等 5 个命令。

四个关键设计

  • 执行与审核分离 —— 执行员只把工作做到「可供审核」，从不自我宣布完成；只有 task-reviewer 明确给出「允许提交」裁决，才会解锁该阶段的 commit。
  • 一次连续运行 —— 澄清与授权全部前置；运行中的决策自主决断并归档到 issues.md 供事后人工审阅，绝不中途停下来问你。
  • 档位而非模型名 —— 计划里只标 T1–T4 能力档位，实际模型在运行时根据平台当下暴露的选项动态匹配，新旗舰会被自动纳入。
  • 冲突域 —— 触碰同一文件 / 配置 / 数据库写入目标 / 端口的任务必须串行；彼此无交集的任务并入同一批次并行跑。

核心概念Agent 团队（11 个）

每个 agent 都是一份带 YAML frontmatter 的系统提示词，其 description 描述「什么时候该用它」——这也是 Claude Code 自动委派的依据。你可以显式点名（「让 architect 评估一下」），也可以放手让主会话按任务性质自动路由。

产品与设计

Agent职责典型触发

architect 系统设计、技术选型、模块拆分、服务间 API 契约、可扩展性规划、架构评审 任务横跨多个模块 / 服务，或引入新基础设施，动手前先请它

质量与效率

Agent职责典型触发
qa-tester 测试计划、单元 / 集成 / e2e 测试编写、边界用例挖掘、回归分析、bug 最小复现 重要改动之后主动验证；或用户报 bug 需要复现时
code-reviewer Diff 评审：正确性、安全、性能、可维护性（只读） 提交重要改动前主动介入
task-reviewer 对抗式完成度审计（只读）：以「未完成推定」逐条核验收标准、检查改动范围、复跑验证命令，裁决 通过 / 未通过 / 外部阻塞，并明确给出「允许提交」与否 长程计划中某个子任务 / 阶段宣称完成、需要提交前的最终裁决时。注意与 code-reviewer 的分工：它审计「照计划完成了没」，而非代码风格
quick-scout 轻量执行员：找文件 / 符号、核实一个事实、改一处配置、跑一条命令。不做架构、不做评审、不写长篇 小而明确的快任务，用它比重型 agent 便宜得多。唯一固定使用快速档模型（haiku 系）的 agent
模型策略 除 quick-scout 外，所有 agent 的 frontmatter 都是 model: inherit（跟随会话主模型）。真正的档位决策由 model-router 在每次派发时做出；长程台账激活期间还有 hook 硬性把关。

核心概念技能（4 个）

技能是可复用的工作流手册，在场景匹配时自动装载。四个技能互相咬合：scheduler 管长程编排，router 管每次派发用什么档位的模型，token-optimizer 管怎么花得省，spec-workflow 管动手前先立契约。

📋 long-task-scheduler — 长程任务调度

解决的问题：超出单次会话的工作（迁移、大重构、功能史诗、批量修复）如果状态只活在对话里，换个会话就断片。此技能把全部状态外化到磁盘台账——「对话只是缓存，台账才是事实」。

批次台账结构

.whet/plan/{YYYYMMDD}-{NNN}-{slug}/
├── plan.md       # 阶段、冲突域、批次、验收标准、档位、恢复点
├── issues.md     # 活跃阻塞、决策、缺口——一切需要人工审阅的东西
├── progress.md   # 每轮收敛记录：状态、提交、下一步
└── {YYYYMMDD}-{主题}-{decision|blocker}.md # 重大决策/阻塞归档（按需）

目录名规则：{YYYYMMDD} 开始日期 + {NNN} 当日序号（从 001 起）+ {slug} 短横线简名。这样按文件名排序即按时间排序。

交互模型：一次连续运行，运行中零打扰

人工介入只发生在两个时点：

  1. 启动前 — 澄清 + 一次性预授权 澄清目标、约束、并发度，并收集一次性预授权清单：本次工程需要的每一个不可逆外部动作和可写目标。问一次，批一次。
  2. 结束后 — 审阅 issues.md 运行中所有自主决策与阻塞都归档在这里，是你事后审计的唯一入口。

运行期间：清单内的动作直接执行，不再请示；清单外的动作绝不执行——记入 issues.md 标记「待人工审阅」，绕过它继续推进。无论是否授权，三条硬禁令始终生效：删除业务数据、泄露机密、伪造验证结果。

编排循环（每一轮）

  1. 对齐事实 —— 读 plan.md / issues.md / progress.md / 最新归档 / git status，绝不凭记忆推断进度；
  2. 选下一批次 —— 按依赖与冲突域：同文件 / 同配置 / 同数据库写入目标 / 同端口 ⇒ 串行；无交集 ⇒ 并行一批。运行中若冲突扩散，回退到串行；
  3. 派发执行员 —— 给每个任务发：任务 ID、精确输入（路径 + 行号范围，而非粘贴整个文件）、禁止事项、验收标准、验证命令、模型档位（作为显式 model 参数传入——台账激活时 hook 会拦截漏传的派发）。执行员只报告「可供审核」；
  4. 对抗式审核 —— 每条完成声明交给 task-reviewer（只读、未完成推定），拿到明确的「允许提交」才解锁 commit；
  5. 收敛 —— 通过 → 更新台账 → 按阶段提交（提交信息带阶段 ID，范围仅限本阶段改动）→ 下一批。未通过但可继续 → 缺口记入 issues.md、重派（重复失败升一档）。外部阻塞 → 归档、标记待人工、推进其余任务；
  6. 循环直到全部阶段完成或仅剩外部阻塞，最后输出收尾简报：各阶段终态、全部提交、汇总的「待人工审阅」清单。

任务纪律

  • 每个任务 ≤ 一个专注会话；必须有可验证的完成标准和恢复点（从这里续跑需要什么为真）；
  • 执行员必须自验（构建 / 测试）并独立重读最终状态——工具的成功提示不算证据；
  • 若运行环境不允许子代理再派子代理，诚实降级为「派发规划器」：产出精确的批次 / 提示词 / 档位清单，由顶层会话逐一执行。绝不虚报「已并行执行」。
反模式（技能会主动规避） 计划状态只活在对话里 ✕ · 没跑验收就说「完成」✕ · 没有审核裁决就提交 ✕ · 中途停下来问预授权清单里已回答的问题 ✕ · 并行派发共享冲突域的任务 ✕ · 一个任务变了就重写整份计划 ✕

🎚️ model-router — 自动模型选择

核心信条：把每件事路由到能可靠完成它的最便宜档位；升档凭证据，不凭感觉。

T1–T4 档位抽象（平台无关，永不硬编码模型名）

档位画像适合成本
T1 强推理 最强推理、长上下文、高风险判断 架构设计、棘手的根因排查、跨模块重构、高风险对抗式审核、长程编排 高
T2 均衡 扎实的编码 + 理解力，性价比最佳 按规格实现功能、写测试、配置工作、常规对抗式审核 中
T3 快速 快、便宜，胜任窄而确定的工作 查找、只读侦察、归档、摘要、模板填充、机械式修改 低
T4 极限 超越 T1；默认禁用，需用户明确同意 T1 都解不开的罕见难题、重大决策升级 最高

运行时选型协议（每次派发前执行）

  1. 探测可用性 —— 读取当前工具实际暴露的模型列表（Claude Code 里是 Agent 工具 model 参数的候选与 /model），绝不凭记忆假设某模型存在或最强；
  2. 按能力分级 —— 依家族定位、推理支持、上下文长度、价格档，把可用模型归入 强推理 / 均衡 / 快轻 三组；
  3. 映射档位 —— T1 → 最强推理模型；T2 → 均衡；T3 → 快轻。若继承会话模型已满足目标档位，则省略 model 参数；
  4. 抗演化 —— 平台上新旗舰自动进入 T1 候选；退役模型向下映射到最近的存活档位；
  5. 诚实降级 —— 平台只支持会话级选型时，把同档任务合批并在提示词中注明推荐档位；完全无法选型时记录档位缺口——能力敏感的任务卡在 T2 以下时标记待人工审阅，而不是低质量硬跑。

路由评分表：三轴取最高

评估轴T3T2T1
推理深度机械 / 模式匹配标准工程问题多约束、含糊、需权衡
爆炸半径错了便宜且明显错了赔一轮评审错了昂贵或静默（迁移、认证、资金、并发、公开 API）
上下文整合单文件一个模块 + 邻居全系统

拿不准时默认 T2 —— 它适合约 70% 的工程任务。

升档与降档

  • 从最低够用档起步。以下情形升一档：审核打回、构建反复失败、根因久攻不下、任务暴露隐藏复杂度（认证 / 资金 / 并发）。每次升档及其触发原因都记入任务台账；
  • 长尾降档：强模型产出一个验证过的样例后，剩余重复实例带着样例用 T3 扇出；
  • 能力敏感的工作绝不为省钱降档：架构决策、安全 / 门禁判断、不变量校验 ≥ T2，高风险的用 T1；
  • 成本敏感的工作绝不盲目升档：只读侦察、归档、对账保持 T3，除非暴露出隐藏复杂度；
  • 每次派发注明 档位 + 实际模型 + 一行理由。
与 token 优化的关系 模型选择主导成本：一枚 T1 token 是 T2 的数倍、T3 的一个数量级以上——但 T3 重试四次也可能比 T1 一次通过更贵。先路由档位，再在选定档位内做 token 优化。

💰 token-optimizer — Token 消耗优化

前提：省 token 绝不以牺牲质量为代价——任何会损害架构判断、门禁验证、根因分析或安全边界的「节省」都被直接否决。便宜的错误答案比昂贵的正确答案更贵。

两种模式

  • 诊断（默认） —— 只读审计，产出带证据的浪费清单，每项附影响估计；
  • 修复（仅在用户明确授权后） —— 应用项目级修复，每改一处验证项目仍可加载 / 构建。

会话内手法（时刻适用）

类别要点
输入最小化 先用索引 / glob 搜索定位，再读命中处 ± 小窗口；传路径 + 行号范围，不粘贴文件正文（尤其不要把同一个大文件同时喂给执行员和审核员）；批量阅读交给子代理，让它返回提炼后的答案——原始文件内容永不进入主上下文（长任务上最大的单项杠杆）；长程状态外化到磁盘，不反复推导。
缓存友好 派发提示词的稳定前缀保持逐字节一致（角色、约束、事实来源清单），易变内容（ID、时间戳）放尾部；同类型、同档位、同前缀的任务合并成一个并行波次，最大化提示词缓存复用（缓存 TTL 约 5 分钟，交互工作尽量成簇进行）。
输出纪律 答案先行，不复述、不预告；改动只发增量，不重写整个文件；用 file:line 引用代替回贴代码；要求子代理精简返回：「结论 + 证据（file:line / 命令），不要散文」。
派发经济学 先路由档位（model-router）；开 N 个 agent 之前自问：一条 grep 能不能解决？每个 agent 拿到的切片是否够窄？返回格式是否最小？评审只看当前 diff，不做全量重扫；查过的结论查归档，不重复调查。

项目级四层诊断

每条发现必须给出：证据（文件 / 配置 / 派发现场）+ 影响估计（高 / 中 / 低）+ 具体修复。没有证据的「感觉可以优化」不算发现。

层范围典型浪费
L1 会话 / 任务每次派发档位错配（T1 干查找、T3 硬啃难题反复返工）；上下文冗余（粘贴大文件、同文件喂多个 agent）；易变内容破坏缓存前缀；重复劳动（重查已定论的结论、全量重扫、失败后同档同提示词重试）。
L2 项目配置每个会话都要付的固定成本常驻注入文件（CLAUDE.md / 规则文件）中的过期内容、重复段落、历史变更记录（最常见的膨胀源——只留「当前结论 + 归档指针」）；agent description 过长（常驻主上下文，应压到 1–2 句触发条件，细节放正文）；hook 输出啰嗦；未使用的 MCP 挂载（工具 schema 是永久上下文）；构建产物 / 依赖 / 生成文件没排除出搜索。
L3 仓库结构信息布局同一事实在 N 处维护（agent 要读 N 次并交叉核对）；归档粘贴仓库里已有的代码（应引用 file:line）；agent 常读的热点大文件（拆分或做索引 / 摘要）。
L4 流程协作方式派发 / 汇报模板带套话（应只有结论 + 证据）；评审范围超出当前改动；同类同档同前缀任务没合批。
质量红线（永不为 token 让步） 架构决策、安全 / 门禁判断、不变量验证、不可逆动作的决策——不降档、不裁上下文；对抗式审核、终态重读、门禁验证——不跳过；任何可能导致静默失败的节省，直接否决并说明原因。

📐 spec-workflow — 规格驱动开发

先写契约，再写代码。一个功能流经三份门控文档，每道门都需要你确认后才能继续：

💡 想法
→
requirements.md做什么 · 为什么
门：每个故事有可测验收标准
→
design.md怎么做
门：覆盖全部验收标准
→
tasks.md执行清单
每项可追溯到 AC / 设计
→
实现 + 验证

各阶段要点

阶段产出委派过门条件
1 需求 requirements.md：问题与动机、用户故事（GIVEN/WHEN/THEN 验收标准）、非目标、约束、待决问题 面向用户的功能交给 product-manager 起草 每个故事都有可测试的验收标准；非目标写明；用户确认。过门之前不做任何设计。
2 设计 design.md：三句话概述、架构（多组件时配 Mermaid 图）、数据模型、API 契约、错误处理与边界、测试策略、被否方案 跨模块工作交给 architect 设计覆盖每一条验收标准；AC 列表中的边界用例都有着落；用户确认。
3 任务 tasks.md：勾选清单，每项 ≤ 半天、可独立验证、按依赖排序 — 每项任务必须能追溯到某条 AC 或设计章节——映射不到的任务直接砍掉。多会话工程把这份清单交给 long-task-scheduler。
4 实现与验证 代码 + 测试 对口的开发 agent 逐项实现并勾选；qa-tester 验证每条 AC 有过测的用例；code-reviewer 对照设计审最终 diff 与设计的偏差要么修掉，要么写回 design.md 并附理由。

什么时候可以跳过

改动类型流程
Bug 修复、小调整直接修（带测试）
单模块功能、需求清晰轻量版：PR 里附验收标准清单 + 任务
多模块功能、新 API、需求含糊完整规格流程
规格随代码版本化 规格保存在 spec/<feature-slug>/，与代码同仓提交。已合并功能的规格即文档；未合并的规格即计划。实现若与设计有偏差，在同一个 PR 里更新规格——绝不让 spec/ 腐烂。

日常使用斜杠命令

五个命令是四个技能的「点火开关」。在 Claude Code 会话中直接输入即可。

/whet:spec <功能描述>

为一个功能启动规格驱动开发。创建 spec/<feature-slug>/，驱动需求 → 设计 → 任务三段门控，每道门都会暂停等你确认；起草工作按技能指引委派给 product-manager 与 architect。

/whet:spec 给站点加一个支持第三方登录的用户中心

/whet:plan <目标描述>

为长程工程创建持久化批次台账并开始编排。流程分三步：

  1. 澄清目标、约束、并发度，收集一次性预授权清单——这是全程唯一可以向你提问的时点；
  2. 从技能模板创建 .whet/plan/{YYYYMMDD}-{NNN}-{slug}/（plan.md / issues.md / progress.md），拆解出阶段、冲突域、验收标准、恢复点与模型档位；
  3. 计划与你确认后，从第一批次开始编排循环。
/whet:plan 把整个后端从 Java 8 + Spring 4 升级到 Java 21 + Spring Boot 3

/whet:resume

从最新的批次台账断点续跑。定位 .whet/plan/ 下最新批次目录，读三份核心文档；低成本复核最近标记完成的任务（文件在不在、测试绿不绿），如与现实不符先校正台账；用 ≤ 5 行汇报「当前阶段 / 最近完成 / 下一动作 / 未决阻塞」，然后继续编排——plan.md 已回答的问题不会再问你。若无台账，会提示改用 /whet:plan 创建。

/whet:review [基准分支]

对当前改动做一次全团队评审：

  1. 用 git 确定 diff 范围（已暂存 + 未暂存 + 领先基准分支的提交；基准分支缺省为默认分支）；
  2. 并行委派 code-reviewer（正确性 / 安全 / 性能 / 可维护性全面评审）与 qa-tester（找出没测的改动逻辑、跑相关测试套件、报告缺口与建议用例）；
  3. 合并两份报告、去重、按严重度排序，产出一份带明确裁决与优先级修复清单的评审。
/whet:review develop

/whet:optimize [范围]

项目级 token 消耗审计（只读）。范围可选 session / config / full（缺省 full）。走完四层诊断，产出「Token Audit」报告：每条发现附具体证据、影响估计与可直接执行的修复，外加会伤害质量而被否决的「伪节省」清单。不做任何改动——呈现排序后的修复清单，等你授权后才进入修复模式。

/whet:optimize config

日常使用Hooks 安全钩子

三个钩子在关键节点自动介入，无需配置，随插件生效：

钩子时机行为
session-ledger SessionStart
（启动 / 恢复会话）		 检测项目里是否存在 .whet/plan/ 批次台账；有则提示最新批次路径，引导会话从台账恢复状态而不是凭记忆。
protect-sensitive PreToolUse
（Write / Edit 之前）		 拦截对敏感文件的意外写入：.env 系列、*.pem / *.key / *.p12 / *.jks 等密钥材料、id_rsa / id_ed25519、credentials、.npmrc / .pypirc。示例 / 模板文件（.env.example 等）放行。确需修改时，由你本人操作或明确批准。
enforce-model-tier PreToolUse
（Task / Agent 派发之前）		 仅在项目存在 .whet/plan/ 台账时生效：派发 whet agent 若未显式传 model 参数则拦截，并回显 model-router 评分表提示重新派发——确保每次派发都是有意识的档位选择，而不是默默继承会话旗舰模型。quick-scout 豁免（其 frontmatter 已固定快速档）；非 whet agent 与无台账的项目完全不受影响。
被 enforce-model-tier 拦下了怎么办？ 这是预期行为，不是故障。按拦截信息里的评分表（推理深度 / 爆炸半径 / 上下文整合）给任务定档，然后带上 model=<家族别名> 重新派发，并按台账要求注明 档位 + 模型 + 一行理由。

日常使用台账文件详解

四份模板位于插件的 skills/long-task-scheduler/templates/，由 /whet:plan 实例化到你项目的批次目录。理解它们的分工，事后审计会轻松很多：

plan.md — 计划（写一次，增量更新）

  • 头部状态栏：批次名、创建时间、整体状态（未开始 / 运行中 / 暂停 / 完成 / 中止）、当前阶段；
  • §0 执行原则：一次连续运行、编排者只派发不亲自实现或审核、执行 → 对抗审核 → 提交、冲突串行无冲突并行、最低够用档起步、重大事项自动归档；
  • §0.1 一次性预授权清单：表格列出每个不可逆动作 / 可写目标及其范围，启动前由你确认——清单外 = 绕行 + 记录；
  • 随后是阶段拆解：每阶段的任务、冲突域、验收标准、模型档位、恢复点。

issues.md — 问题与决策（每轮更新）

  • 问题总表：编号（L1、L2…）、状态（☐ 未决 / ▶ 处理中 / ✓ 已解决 / ⚠ 待人工审阅 / ⊘ 跳过）、所属阶段、负责人、决议、归档链接；
  • 「待人工审阅」小节是你跑完后的唯一审计入口：每个自主决策、档位升级、被绕行的清单外动作、外部阻塞都在这里；
  • 已解决问题保留一行摘要，细节进归档文件。

progress.md — 进度（每轮更新）

  • 顶部摘要：当前阶段、整体状态、已提交的阶段、未决阻塞、已归档决策；
  • 逐轮记录（Round N）：本轮派发的批次（任务 / 档位 / agent / 状态）、审核裁决、提交哈希、token 优化动作、阻塞与决策。

归档文件 — 重大决策 / 阻塞（按需）

命名 {YYYYMMDD}-{主题}-{decision|blocker}.md，内容包括：背景、决策（或阻塞描述）、被否选项及原因、风险、需人工确认的要点、影响。可复原的决策立即生效；预授权清单之外的不可逆事项只能落为「绕行 + 记录」。

台账是你的数据 .whet/plan/** 在你的仓库里，插件更新不会碰它。建议将其纳入版本控制——它同时是项目决策史。若不想入库，在项目 .gitignore 中排除即可。

附录版本与兼容性

Whet 遵循 SemVer，main 分支即分发渠道、始终保持可安装，发布打 vX.Y.Z 标签：

级别含义对你的影响
MAJOR 破坏性：重命名 / 删除 agent、skill、command；变更 .whet/plan/ 台账结构；收紧 hook 拦截规则 changelog 必附迁移说明。惯例是旧数据继续可读（例如旧台账继续工作，仅新批次采用新结构）。
MINOR 新增 agent / skill / command 与能力、模板补充 向后兼容，直接升级。
PATCH 提示词调优、措辞与文档修正 无感升级。

兼容性承诺：MINOR / PATCH 版本绝不重命名 agent / skill / command（你的肌肉记忆和项目文档都引用着它们）；已落盘的台账文件是用户数据，新版技能必须继续兼容旧结构，否则明示迁移路径。

附录常见问题

Q：所有任务都要走 /whet:spec 或 /whet:plan 吗？

不需要。修 bug、小调整直接说就行。单模块、需求清晰的功能用轻量版（PR 里附验收标准）。只有多模块功能、新 API、需求含糊、或超出单会话的工程才值得完整仪式——技能文档里明确写了「什么时候跳过」。

Q：task-reviewer 和 code-reviewer 有什么区别？

code-reviewer 审代码：正确性、安全、性能、可维护性。task-reviewer 审完成度：对照计划核验收标准、查改动范围、复跑验证，以「未完成推定」的对抗姿态给出「允许提交」与否的裁决。长程调度里两者都会出场，但把关阶段提交的是后者。

Q：为什么派发 agent 时被 hook 拦截了？

说明你的项目有活跃的 .whet/plan/ 台账，且这次派发没有显式指定 model。这是 enforce-model-tier 守卫的预期行为：按拦截信息里的评分表定档，加上 model 参数重新派发即可。项目里没有台账时此守卫完全不生效。

Q：长程任务中途我想改需求怎么办？

随时可以介入——台账在磁盘上，停下来改 plan.md（增量修改，不要整篇重写），然后 /whet:resume 继续。调度器每轮开始都会重读台账对齐事实，你的修改自然生效。

Q：T4 档位怎么启用？

T4 默认禁用，只有你明确同意才会使用——要么在 /whet:plan 的预授权清单中批准，要么运行外单独授权。它面向 T1 都解不开的罕见难题，日常几乎用不到。

Q：Whet 会不会自己动我的 .env 或密钥文件？

不会。protect-sensitive 钩子在写入前拦截所有敏感文件（env、密钥、证书、凭据），只放行 .example / .sample / .template 后缀的模板。确需修改时会请你本人操作或明确批准。此外长程运行还有三条与授权无关的硬禁令：不删业务数据、不泄露机密、不伪造验证结果。

Q：换了台电脑 / 新开会话，长程任务还在吗？

在。所有状态都在 .whet/plan/ 的文件里（如果你把它提交进了 git，克隆下来就有）。新会话启动时 session-ledger 钩子会自动提示最新批次；直接 /whet:resume 即可续跑。

Q：如何贡献或报告问题？

欢迎 Issue 与 PR：github.com/mufasa007/whet。提 PR 前请用 claude --plugin-dir . 本地验证、bash -n hooks/scripts/*.sh 检查脚本语法，并遵循仓库 CLAUDE.md 中的约定（尤其：agent description 写清触发时机、绝不硬编码模型名、文档保持 token 精简）。
